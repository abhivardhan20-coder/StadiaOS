### H1. "Logical decision-making" currently lives entirely inside an LLM prompt â€” make it deterministic

**Why it matters:** The README claims OpsCopilot "evaluates incoming live telemetry against predetermined thresholds" and escalates vs. suggests. Looking at `server.ts`, that logic doesn't exist in code â€” it's a sentence inside the Gemini `systemInstruction`. That means the "logical decision making based on user context" requirement (a named Challenge Expectation, not just a nice-to-have) is entirely dependent on the model's mood, isn't testable, and isn't auditable. This is the single highest-leverage fix for the "smart, dynamic assistant" criterion.

**Prompt to use:**
> "In `server.ts`, extract a pure, synchronous function `evaluateOpsPolicy(liveContext: LiveContext): { level: 'normal' | 'suggest' | 'escalate'; reasons: string[]; suggestedActions: string[] }` that applies explicit thresholds (e.g. any zone density â‰¥ 85 â†’ escalate, â‰¥ 75 â†’ suggest, any `securityIncidents` entry â†’ escalate). Call it inside the `/api/ops-copilot` handler before hitting Gemini, inject the resulting `level`/`reasons` into the systemInstruction so the model's language matches the deterministic decision instead of inventing it, and return `policy` alongside the streamed text so the frontend can render an 'ESCALATED'/'SUGGESTED' badge without parsing markdown. Add unit tests in `server.test.ts` covering normal/suggest/escalate/multiple-breach cases."

**Snippet (drop into `server.ts` above the `/api/ops-copilot` route):**
```ts
export interface OpsPolicyResult {
  level: "normal" | "suggest" | "escalate";
  reasons: string[];
  suggestedActions: string[];
}

const DENSITY_ESCALATE = 85;
const DENSITY_SUGGEST = 75;

/**
 * Deterministic escalation policy. Kept separate from the LLM call so the
 * decision is testable, auditable, and never depends on model output.
 */
export function evaluateOpsPolicy(liveContext?: { zones?: { name: string; density: number }[]; alerts?: string[] }): OpsPolicyResult {
  const reasons: string[] = [];
  const suggestedActions: string[] = [];
  let level: OpsPolicyResult["level"] = "normal";

  for (const zone of liveContext?.zones ?? []) {
    if (zone.density >= DENSITY_ESCALATE) {
      level = "escalate";
      reasons.push(`${zone.name} density at ${zone.density}% (>= ${DENSITY_ESCALATE}%)`);
      suggestedActions.push(`Dispatch security/medical to ${zone.name}`);
    } else if (zone.density >= DENSITY_SUGGEST && level !== "escalate") {
      level = "suggest";
      reasons.push(`${zone.name} density at ${zone.density}% (>= ${DENSITY_SUGGEST}%)`);
      suggestedActions.push(`Consider opening overflow route near ${zone.name}`);
    }
  }

  if ((liveContext?.alerts ?? []).length > 0) {
    level = "escalate";
    reasons.push(...(liveContext?.alerts ?? []));
  }

  return { level, reasons, suggestedActions };
}
```

```ts
...`;
```

### H3. Google OAuth flow has no `state` parameter â€” CSRF gap on login

**Why it matters:** In `server.ts`, `/api/auth/google` builds the authorize URL without a `state` param, and `/api/auth/google/callback` never checks one. This is a textbook OAuth CSRF vector (an attacker can trick a victim's browser into completing an attacker-initiated auth flow) and it's exactly the kind of thing a security-focused review will flag on a submission that otherwise has helmet/CORS/rate-limiting in place.

**Prompt to use:**
> "In `server.ts`, generate a cryptographically random `state` value in `/api/auth/google`, store it in an httpOnly cookie (`oauth_state`, short-lived, `secure`, `sameSite: 'lax'`), and include it as the `state` query param sent to Google. In `/api/auth/google/callback`, compare the returned `state` query param against the cookie value and reject the request (clear message, no token issuance) if they don't match or either is missing. Clear the `oauth_state` cookie after a successful or failed callback."

**Snippet:**
```ts
import crypto from "crypto";

app.get('/api/auth/google', (req, res) => {
  const state = crypto.randomBytes(16).toString('hex');
  res.cookie('oauth_state', state, { httpOnly: true, secure: true, sameSite: 'lax', maxAge: 5 * 60 * 1000 });

  const protocol = req.headers['x-forwarded-proto'] || req.protocol;
  const host = req.get('host');
  const redirectUri = `${protocol}://${host}/api/auth/google/callback`;
  const url = `https://accounts.google.com/o/oauth2/v2/auth?client_id=${GOOGLE_CLIENT_ID}&redirect_uri=${encodeURIComponent(redirectUri)}&response_type=code&scope=email%20profile&access_type=offline&state=${state}`;
  res.redirect(url);
});

app.get('/api/auth/google/callback', async (req, res) => {
  const { code, state } = req.query;
  const expectedState = req.cookies?.oauth_state;
  res.clearCookie('oauth_state');

  if (!expectedState || state !== expectedState) {
    console.warn('OAuth state mismatch â€” possible CSRF attempt.');
    return res.send(`<html><body><script>
      window.opener.postMessage({ type: 'OAUTH_ERROR', reason: 'state_mismatch' }, window.location.origin);
      window.close();
    </script></body></html>`);
  }
  // ...existing token exchange using `code`
});
```

### M2. Rate-limit / trust-proxy mismatch

**Why it matters:** `app.set("trust proxy", 1)` tells Express to trust `X-Forwarded-For`, but `apiLimiter` is configured with `validate: { trustProxy: false, xForwardedForHeader: false }`. Behind Cloud Run (which the README says this targets), every request arrives through a proxy â€” with this combination the rate limiter can end up keying on the wrong IP (or throwing/ignoring proxy headers), meaning your 30 req/min limit may not actually apply per-client in production. That's a real gap for both **Security** and **Efficiency** (resource protection).

**Prompt to use:**
> "Align the rate limiter's trust-proxy validation with the Express `trust proxy` setting so IP-based limiting is accurate behind Cloud Run's single reverse proxy hop. Add a test that sends requests with a spoofed `X-Forwarded-For` and confirms the limiter isn't bypassed."

**Snippet:**
```ts
const apiLimiter = rateLimit({
  windowMs: 60_000,

  standardHeaders: true,
  legacyHeaders: false,
  message: { error: "Too many requests, please try again later." },
```
